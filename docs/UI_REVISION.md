# Ревизия интерфейса NekoMiFlash

Полная инвентаризация UI-слоя и план реорганизации. Документ для принятия решений —
ничего в коде пока не меняется.

## 1. Что есть сейчас (факты)

- **80 кнопок**, 7 табов, 16 текстовых полей на одном экране.
- Все 80 кнопок подключены в коде (мёртвых нет — чистить нечего, задача в реорганизации).
- Навигация — 7 табов, но в UI видно только 4 (Home / Fastboot / ADB / Files);
  Diagnostics / Reports / Help открываются как скрытые страницы.

### Распределение кнопок по страницам

| Страница | Кнопок | Что внутри |
|---|---:|---|
| **pageHome** | 22 | refresh, getvar, 4× профиль безопасности, 6 плиток, 5× operation center, 3× reboot, service, language |
| **containerFastboot** | 29 | 5× flash раздел, 5× Xiaomi ROM, 8× Recovery, 4× Magisk, 7× очередь |
| **containerAdb** | 1 | sideload |
| **pageFiles** | 3 | import, analyze, copy path |
| **pageDiagnostics** | 13 | refresh, self-test ×3, reports, autoscan, debug, battery, permissions, forum, export/clear log, help |
| **pageOperation** | 2 | console, stop |
| **pageHelp** | 4 | 3× wizard, console |
| **consolePanel** | 3 | history up/down, send |

## 2. Главные проблемы (UX-аудит)

### 2.1. Перегрузка pageHome (22 кнопки на «главной»)
Главный экран должен показывать 3–5 основных действий. Сейчас на нём свалено всё:
профиль безопасности (4 кнопки-переключателя), operation center (5 кнопок), reboot (3),
плюс служебные (language, service). Это и есть та «куча», о которой речь.

### 2.2. Дублирование одних и тех же действий
Одно действие живёт в 3–6 местах одновременно:

| Действие | Сколько кнопок | Где |
|---|---:|---|
| Консоль/Терминал | 4 | tile, home, operationCenter, operation |
| Reports/Отчёты | 6 | tile, operationCenter, openFolder, forum, selfTestForum, operationCenterForumZip |
| Reboot | 5 | system, bootloader, recovery + recovery-page ×2 |
| Flash boot | 4 | flash, recovery, magisk, queue |
| Flash init_boot | 4 | те же 4 контекста |
| Log (export/clear/debug) | 4 | export, clear, debug, operationCenterLog |
| Cancel/Stop | 3 | cancel, operationStop, operationCenterCancel |

Это не всегда плохо (одно действие в разных контекстах), но Reports ×6 и Reboot ×5 —
явный перебор, их стоит свести в один компонент.

### 2.3. Reboot-кнопки разбросаны
3 на Home + 2 на Recovery-секции = 5 кнопок перезагрузки в разных местах.
Логично собрать в один блок «Reboot →» с подпунктами (system / bootloader / recovery / fastbootd).

### 2.4. Reports-функции размазаны (6 кнопок)
Открыть папку, форум-отчёт, self-test форум-отчёт, operation-zip, лог — всё про «отчёты»,
но раскидано по Home, Diagnostics и Operation Center. Просится единый экран «Отчёты».

### 2.5. Профиль безопасности занимает место на Home
4 кнопки-переключателя (Novice/Standard/Expert/HighRisk) + описание — это настройка,
а не частое действие. Место ей в настройках/«Ещё», а не на главной.

## 3. Предлагаемая структура (declutter)

### ГЛАВНАЯ — только то, ради чего открывают приложение
```
[ Статус устройства: ● режим · модель · слот ]   [⟳] [⋯]

   🐱 Прошить ROM        (Xiaomi ROM / разделы)
   🩹 Recovery / Root    (bootloop, Magisk)
   📂 Файлы              (импорт образов)

[ ▾ CONSOLE — свёрнута ]
```
Три крупные кнопки + статус + свёрнутая консоль. Всё.

### «⋯ Ещё» (BottomSheet, выезжает снизу) — второстепенное
```
Reboot →     system / bootloader / recovery / fastbootd
Отчёты →     папка / форум-отчёт / экспорт лога
Сервис →     self-test / диагностика / autoscan / debug-лог
Настройки →  профиль безопасности / язык / разрешения / батарея
Помощь →     wizard'ы / справка
```

### Вкладки Fastboot / ADB — оставить как есть
Там профессиональные функции (flash разделов, Xiaomi ROM, очередь, sideload) —
их трогать не нужно, кто дошёл до вкладки, тот знает что делает.

## 4. Что НЕ трогаем
- Логику ADB/Fastboot/USB протоколов.
- Все 80 обработчиков кнопок — ID сохраняются, просто перемещаются в layout.
- Вкладки Fastboot/ADB/Files (профессиональный слой).

## 5. Порядок внедрения (по шагам, каждый — отдельная сборка/тест)
1. **Reboot-меню**: 5 reboot-кнопок → один BottomSheet «Reboot». (низкий риск)
2. **Отчёты-меню**: 6 reports-кнопок → один экран/BottomSheet «Отчёты». (низкий риск)
3. **Профиль безопасности + настройки** → в «Настройки» BottomSheet. (средний риск — там состояние)
4. **Главная → 3 кнопки**: убрать с Home всё, кроме 3 главных + статус + консоль,
   остальное — в «⋯ Ещё». (высокий риск — много перемещений)

Каждый шаг сохраняет все ID (код не ломается), меняется только расположение в layout
и добавляется BottomSheetDialog. Тестировать после каждого шага.

## 6. Риск-напоминание
Сборку нельзя скомпилировать/протестировать в среде разработки — только через
цикл Termux→GitHub Actions→установка. Поэтому шаги маленькие и проверяемые,
а не «всё разом». BottomSheet — стандартный Android-компонент, низкий риск,
но состояние профиля безопасности (шаг 3) требует аккуратности.
