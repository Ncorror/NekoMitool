## 5. Идеи для стабильности (приоритет)

### 5.1. Operation Center показывать только во время операции (UX, не сделано)
Сейчас висит пустым с «ОЖИДАНИЕ». Логично скрывать, когда операции нет.
Низкий риск. Это последняя невыполненная UX-идея из ранних обсуждений.

### 5.2. FlashOperationService для лёгких операций (стабильность)
Кнопка «Обновить» (getvar) поднимает foreground-сервис, что на Android 14
провоцировало краш (уже пофикшен через stopSelf, но первопричина осталась).
Правильно: не поднимать сервис для лёгких операций (getvar/diagnostics),
только для реальной прошивки. Средний риск (трогает startOperation).

### 5.3. Декомпозиция MainActivity (поддерживаемость)
3061 строк в одном файле. Отложенный план: вынести FlashDialogs (диалоги
flash/preflight/confirm), UsbReceiver (приём USB + разрешения — самый рискованный),
Xiaomi use-case (из DeviceViewModel). Каждое — свой цикл. Это не «мусор»,
а долг по поддерживаемости. Можно не трогать, пока работает.

## 6. Идеи для разгрузки интерфейса

### 6.1. Очередь прошивки (flash queue) — нужна ли на главном виду?
Блок очереди занимает место на Fastboot. Если используется редко — можно
свернуть в раскрывающуюся секцию или меню.

### 6.2. Sideload-вкладка (ADB) — одна кнопка
containerAdb содержит всего 1 кнопку (sideload). Целая вкладка ради одной
кнопки — кандидат на объединение или перенос.

### 6.3. Diagnostics-страница (13 кнопок) — проверить актуальность
Скрытая страница диагностики содержит много кнопок. Стоит проверить,
все ли реально нужны после реорганизации (часть могла продублироваться в меню).

## 7. Рекомендованный порядок (если продолжать)

1. **Чистка мусора** (orphan-метод + 2 drawable) — 5 минут, нулевой риск.
2. **Operation Center только при операции** — завершить последнюю UX-идею.
3. **FlashOperationService для лёгких операций** — убрать первопричину краша «Обновить».
4. (Опционально) Декомпозиция MainActivity — долг по поддерживаемости.
